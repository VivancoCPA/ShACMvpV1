## Context

`calcularKpis(periodo)` in `shc-controldoc/src/mocks/handlers/dashboard.handlers.ts` is always invoked with `currentPeriodo()` — `new Date().toISOString().slice(0, 7)`, i.e. the real device month — from every dashboard handler (`buildJefeCalidadData`, `buildSupervisorData`, `buildAltaDireccionData`, and the standalone `/api/dashboard/kpis` handler). Five of the nine KPI formulas filter their inputs by this period:

| KPI | Filter | Field(s) that must land in the period |
|---|---|---|
| KPI-01 (`calcularKpi01`) | `qe.estado ∈ {CERRADO, VERIFICADO}` AND `fechaCierre` in `monthRange(periodo)` | `fechaCierre` |
| KPI-02 (`calcularKpi02`) | same as KPI-01 | `fechaCierre` |
| KPI-03 (`calcularKpi03`) | same filter, but `quarterInfo(periodo)` window, then `ciclo > 1` | `fechaCierre` (+ `ciclo`) |
| KPI-05 (`calcularKpi05`) | AC `estado === 'CERRADA'` (QE/NC origin) whose parent has `resultadoVerificacion` set AND parent verification date in `monthRange(periodo)` | `fechaVerificacionRealizada` (QE) / `fechaVerificacion` (NC) |
| KPI-07 (`calcularKpi07`) | `qe.auditTrail` entry with `accion === 'ESTADO_CAMBIADO' && estadoNuevo === 'ANALISIS_COMPLETADO'`, timestamp in `monthRange(periodo)` | that audit entry's `timestamp` |

None of the 20 fixtures in `quality-events.fixtures.ts` currently has these date fields inside the real current month (2026-07 as of this change) or quarter (2026-Q3), even though several fixtures already have the *fields themselves* populated (e.g. `qe-2026-004` already has `fechaCierre`, `resultadoVerificacion`, and an `ANALISIS_COMPLETADO` audit entry — just all dated 2025-11 through 2026-02). That is the actual root cause: **period misalignment, not missing fields.** KPI-04/06/08/09 don't hit this problem because KPI-06/08 aren't period-filtered at all, and KPI-04/09 already have fixture data (`kpi04AnioAnteriorFixtures`, `fechaHoraReporte` spread across the "last 6 months ending on reference date 2026-07-01" per the existing `quality-event-msw-fixtures` spec) that happens to reach the current month.

This confirms the existing convention in this codebase: mock fixtures are anchored to a **fixed reference "now"** that gets bumped forward periodically (see `kpi04AnioAnteriorFixtures`'s comment: "Mismo rango que incidents.fixtures.ts (2026-01 a 2026-07)"), not derived from `Date.now()` at fixture-definition time. This change follows that same convention rather than introducing a new one.

## Goals / Non-Goals

**Goals:**
- Make KPI-01, KPI-02, KPI-03, KPI-05, KPI-07 compute non-zero, non-100% values against the real current period (2026-07 / 2026-Q3) using only edits to existing fixtures.
- Keep every edited fixture internally coherent: `fechaHoraReporte` ≤ audit-trail timestamps in creation order ≤ `fechaCierre` ≤ `fechaVerificacionRealizada`, consistent with RN-QE-002 (causa raíz approved before `EN_EJECUCION`) and RN-QE-004 (dual signature on cierre).
- Produce a realistic mix per KPI (not all "en plazo", not all "fuera de plazo"; not all "efectivo").

**Non-Goals:**
- Changing `calcularKpis`, any KPI formula, or the fact that the period defaults to `currentPeriodo()`. A "pin the dashboard to a fixed demo period" change is a separate concern or possibly config-driven — deliberately out of scope here, since introducing a period-override flag would be a wider architectural change.
- Fixing the unrelated latent gaps noticed during investigation: `horasTrabajadasFixtures` has no `2026-07` entry (would zero KPI-04's numerator/denominator this month too) and `qe-2026-020` is soft-deleted (`deletedAt` set) yet still counted by `getQeStore()` in dashboard aggregation. Neither was reported as broken by the browser verification that triggered this change, and both are pre-existing conditions unrelated to the 5 KPIs in scope. Flagged here for awareness only.
  - **Confirmed during task 5.3 browser verification**: KPI-04 (Índice de frecuencia de incidentes) renders `0.00` against the post-fix seed as `jefe.calidad@shac.pe`, unlike KPI-05 (also period-gated) which moved from `0.0%` to a real `50.0%` with this same change. Root cause matches the hypothesis above: `calcularKpi04` (`dashboard.handlers.ts`) divides by `horasTrabajadasFixtures` filtered to `periodo === currentPeriodo()`; since `horasTrabajadasFixtures`'s `PERIODOS` list stops at `2026-06` and `currentPeriodo()` is `2026-07` as of this change, the denominator is always `0` and the function short-circuits to `0` regardless of `incidentes` data — not necessarily "zero incidents in period" as the number alone would suggest. Left unfixed here per the Non-Goal above (out of scope: `horasTrabajadasFixtures` is not a `quality-events.fixtures.ts` concern). A future spec extending `horasTrabajadasFixtures.PERIODOS` (same maintenance pattern as `kpi04AnioAnteriorFixtures`) is needed to make KPI-04 period-correct again.
- Adding new fixtures or new fields to the `QualityEvent`/`AccionCorrectivaQE` types.

## Decisions

**1. Reuse existing lifecycle-complete fixtures instead of adding new ones.** `qe-2026-004` (VERIFICADO, BAJA, EFECTIVO) and `qe-2026-009` (CERRADO, BAJA, no verification yet) already have every field KPI-01/02/07 need — they're just dated in 2025-11/2025-12/2026-01. Shifting their existing timestamps (rather than fabricating new fixtures) keeps the fixture count and enum-coverage guarantees in `quality-event-msw-fixtures` untouched and minimizes diff size.
  - *Alternative considered*: add 2-3 brand-new fixtures dated in the current period. Rejected — grows the fixture set past what `document-list-view`/pagination-oriented specs assume, and duplicates data already present in a slightly-wrong location in time.

**2. Use `qe-2026-015` (already `ciclo: 2`) for the KPI-03 reincidence case, promoted from `PENDIENTE_CIERRE` to `CERRADO`, instead of bumping `ciclo` on a fresh record.** It already carries a coherent reincidence narrative ("Reincidencia detectada en auditoría de seguimiento") and just needs one more state transition (`PENDIENTE_CIERRE` → `CERRADO`) plus `fechaCierre` and `cierreFirmaSupervisorId` (RN-QE-004 dual signature, currently missing on this fixture) to become a `CERRADO` record in-period.
  - *Alternative considered*: bump `ciclo` on `qe-2026-010` (VERIFICADO) to 2 and fabricate a reapertura audit entry. Rejected — `qe-2026-010`'s existing narrative has no reapertura context, so inventing one is a bigger, less-grounded edit than finishing `qe-2026-015`'s already-in-progress reincidence story.
  - *Alternative considered*: reuse `qe-2026-005` or `qe-2026-017` (both `REABIERTO`, `ciclo: 2`, already the source of the KPI-05 NO_EFECTIVO case per decision 4). Rejected for KPI-03 — `REABIERTO` is excluded by `qeCerradosEnPeriodo`'s `estado ∈ {CERRADO, VERIFICADO}` filter, and pushing either through a full second investigation cycle to a real re-close is a much larger, narratively risky edit than promoting `qe-2026-015`.

**3. `qe-2026-009`'s new `ANALISIS_COMPLETADO` audit entry does double duty as the "fuera de plazo" KPI-01/02 case.** No separate fixture is needed for KPI-07 — `qe-2026-009` already becomes a `CERRADO`-in-period record for KPI-01/02, and one added audit entry (dated before its new `fechaCierre`) gives KPI-07 a data point in the same edit.

**4. KPI-05's NO_EFECTIVO case reuses `qe-2026-017`'s existing `resultadoVerificacion: 'NO_EFECTIVO'` without changing its `estado`.** `calcularKpi05` reads `qe.resultadoVerificacion` / `qe.fechaVerificacionRealizada` and the QE's own `accionesCorrectivas`, with no filter on the QE's current `estado`. `qe-2026-017` is `REABIERTO` (ciclo-1 verification already failed — that's exactly what `resultadoVerificacion: 'NO_EFECTIVO'` represents), so adding `fechaVerificacionRealizada` in-period and one `CERRADA` AC to the `qeAccionesCorrectivas['qe-2026-017']` map produces a valid NO_EFECTIVO data point without touching its `estado` or contradicting its narrative ("Las ACs del ciclo 1 ... resultaron NO_EFECTIVO" — describes a closed-but-ineffective AC).
  - *Alternative considered*: flip `nc-012`'s `resultadoVerificacion` to NO_EFECTIVO. Rejected — its AC's `descripcionEvidencia` ("Sensor reemplazado. Certificado de calibración adjunto.") explicitly describes a successful fix; changing the verdict would contradict the evidence text.
  - *Alternative considered*: give `qe-2026-009` (already reused for KPI-01/02/07) a `resultadoVerificacion` too. Rejected — verification only applies after `EN_VERIFICACION`, and pushing `qe-2026-009` there would require moving it out of `CERRADO` and out of `qeCerradosEnPeriodo`'s eligible set, undoing decision 1.

**5. `qe-2026-004` supplies the EFECTIVO counterpart for KPI-05** by adding an entry to `qeAccionesCorrectivas['qe-2026-004']` (currently absent from the map) and moving its existing `fechaVerificacionRealizada` into the current period alongside the rest of its shifted timeline. It already carries `resultadoVerificacion: 'EFECTIVO'`.

## Risks / Trade-offs

- [Real-world date drift] `currentPeriodo()` tracks the actual device clock, so this fix stops producing non-trivial KPI values once the real month rolls past the anchor used here → Mitigation: none needed *in this change* (matches the existing, accepted maintenance pattern for `kpi04AnioAnteriorFixtures`); documented in the proposal's Impact section so a future bump isn't a surprise.
- [Cross-test coupling] `qe-2026-004`, `-009`, `-015`, `-017` may be referenced by id/date in other tests (`dashboard.handlers.test.ts`, `quality-events.fixtures` tests, `ACsPorVencerWidget`/semaforo tests) → Mitigation: run the full fixture/handler/dashboard test suite after editing (already a task below); the semaforo/ACs-por-vencer widgets read `accionesCorrectivas`/`plazoFecha` on *open* ACs, and none of the newly-added ACs are left `PENDIENTE`/`EN_EJECUCION`, so they shouldn't newly appear in "ACs por vencer."
- [`qe-2026-015` dual signature] Promoting `qe-2026-015` to `CERRADO` without `cierreFirmaSupervisorId` would leave it inconsistent with RN-QE-004 as modeled on every other `CERRADO`/`VERIFICADO` fixture → Mitigation: task list explicitly adds `cierreFirmaSupervisorId` when promoting it.
- [Business-day margins] Exact `contarDiasHabiles` weekday counts weren't hand-verified to the day → Mitigation: the chosen date gaps (~17 business days vs. a 22-day BAJA plazo; ~10 weeks vs. a 10-day ALTA plazo; ~10 weeks vs. a 22-day BAJA plazo) have wide enough margins that a few days of weekend miscounting cannot flip the en-plazo/fuera-de-plazo classification.
- **[Gap conocido, no bloqueante]** KPI-04 (Índice de frecuencia de incidentes) sigue en `0.00` tras este fix, a diferencia de KPI-05 que sí pasó a un valor real (`50.0%`). No es un defecto de este cambio — `horasTrabajadasFixtures.ts` (fuera de alcance, ver Non-Goals) no tiene entrada para `2026-07`, así que `calcularKpi04` siempre divide entre `0` horas y retorna `0` sin importar cuántos incidentes con lesionados haya en el período. El `0.00` actual no debe leerse como "cero incidentes reales" — es un artefacto de la desalineación de período entre `horasTrabajadasFixtures` y el mes real. Mitigación: ninguna en este cambio (explícitamente fuera de alcance); queda documentado aquí para diagnóstico futuro. Fix correcto: extender `horasTrabajadasFixtures.PERIODOS` para incluir el mes actual (mismo patrón de mantenimiento ya aceptado para `kpi04AnioAnteriorFixtures`), en una spec futura dedicada a ese fixture.

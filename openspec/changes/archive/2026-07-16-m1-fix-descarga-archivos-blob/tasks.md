**Post-implementation fix not in the original task list:** `useGetArchivoOriginalUrl` and `useGetArchivoDistribucionUrl` needed a `queryClient.invalidateQueries({ queryKey: QUERY_KEYS.documents.detail(documentoId) })` call in `onSuccess` (matching the pattern already used by `useChangeStatus`/`useSignDocument` etc.) so the new `DESCARGA` audit entry appears in the Audit Trail tab without a manual page refresh. Discovered via live-browser verification of 5.3.

**Verification blocker discovered (separate, pre-existing bug, not fixed here):** `getUserFromRequest` in `documents.handlers.ts` (and the identical helper in `dashboard.handlers.ts`) parses the Bearer token with `/^mock-access-token-(.+)-\d{13}$/`, which does not account for the `refreshed-` prefix that `auth.handlers.ts` adds to tokens issued by `POST /api/auth/refresh` (that file's own parser already has the fix: `/^mock-access-token-(?:refreshed-)?(.+)-\d{13}$/`). Since the app performs an automatic refresh shortly after login in normal usage, any role-gated request to `documents.handlers.ts` (e.g. `archivo-original`) ends up treating the user as unauthenticated → falls back to `OPERARIO` → gets a false 403, regardless of actual role. This blocked live-browser verification of 5.1/5.4 for the archivo-original flow specifically (worked around with a Playwright network-level token rewrite for the automated check). Recommended fix: apply the same `(?:refreshed-)?` regex adjustment to `documents.handlers.ts:33` and `dashboard.handlers.ts:63`. Out of scope for this change — flagged for separate follow-up.
